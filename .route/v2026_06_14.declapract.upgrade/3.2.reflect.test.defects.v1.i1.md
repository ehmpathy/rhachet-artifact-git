# 3.2.reflect.test.defects.v1.i1

## defect classifications

### defect 1: ES2022 lib for Error cause

**classification**: practice bug

**why**: the declapract upgrade added `{ cause: error }` syntax to error handlers (jest.integration.env.ts), but did not ensure tsconfig.json has ES2022 lib. this should be part of the practice — if `{ cause }` is used, ES2022 lib must be present.

**citation**: https://github.com/ehmpathy/declapract-typescript-ehmpathy/blob/main/src/practices/tests/best-practice/jest.integration.env.ts

**action**: report to declapract-typescript-ehmpathy — jest.integration.env.ts practice should require ES2022 in tsconfig.json lib array

### defect 2: HelpfulError static .throw() type mismatch

**classification**: repo quirk

**why**: this repo had custom error subclasses that used the static `.throw()` pattern. the upgrade bumped helpful-errors which changed type constraints. fix was to switch to `throw new Error()` pattern. this is a repo-specific migration, not a practice issue.

**action**: none — repo-specific fix already applied

### defect 3: RefByUnique type mismatch in tests

**classification**: repo quirk

**why**: this repo used `refByUnique(mockResult)` helper in tests. the upgrade bumped domain-objects which changed type inference. fix was to use explicit object literals. this is a repo-specific test pattern issue.

**action**: none — repo-specific fix already applied

### defect 4: dpdm circular dependency detection in node_modules

**classification**: adoption candidate

**why**: the `test:lint:cycles` command added by declapract is correct. it detected real cycles in transitive deps that would affect consumers. the fix was to bump upstream deps to versions without cycles. this is the intended behavior.

**action**: none — lint:cycles works as designed. the detection forced us to fix the dep tree, which is the right outcome.
