# 3.1.repair.test.defects.v1.i1

## defect inventory

### defect 1: ES2022 lib for Error cause

**what**: TypeScript error TS2554 — Error constructor with `{ cause }` option requires ES2022 lib

**how** (diff against origin/main): jest.integration.env.ts added `{ cause: error }` to Error constructor

**why**: ES2022 introduced ErrorOptions with `cause` property. tsconfig.json had `lib: ["DOM"]` only — no ES2022 ErrorOptions type.

**fix**: added `"ES2022"` to tsconfig.json lib array (done in stone 2 hazard detection phase)

### defect 2: HelpfulError static .throw() type mismatch

**what**: TypeScript error TS2684 — `this` context type incompatibility when calling `ArtifactAccessDeniedError.throw()`

**how** (diff against origin/main): helpful-errors 1.5.3 → 1.7.3 changed generic signature on `.throw()` static method

**why**: helpful-errors 1.7.3 uses complex `this` type constraints for the static `.throw()` method. subclasses must match the HelpfulError constructor signature exactly. the generic type parameter propagation is stricter.

**fix**: changed from `ArtifactAccessDeniedError.throw(...)` to `throw new ArtifactAccessDeniedError(...)` pattern. this avoids the static method type constraint issue while maintaining the same behavior.

### defect 3: RefByUnique type mismatch in tests

**what**: TypeScript error TS2741 — `refByUnique(mockResult)` return type incompatible with `RefByUnique<typeof GitFile>`

**how** (diff against origin/main): domain-objects 0.31.9 → 0.31.13 changed Ref type inference

**why**: `refByUnique()` function returns a generic type that no longer matches `RefByUnique<typeof GitFile>` exactly. the type constraint became stricter.

**fix**: replaced `refByUnique(mockResult)` with explicit ref object `{ uri: tmpUrl }` in test files. this is clearer and avoids the helper function type mismatch.

files fixed:
- `src/domain.operations/file/gitFileGet.test.ts`
- `src/domain.operations/file/gitFileSet.test.ts`

### defect 4: dpdm circular dependency detection in node_modules

**what**: lint failure — dpdm detects circular dependencies in transitive dependencies (helpful-errors@1.5.3, domain-objects@0.31.9)

**how** (diff against origin/main): upgrade added `test:lint:cycles` command with dpdm and `.dpdmrc.yaml` config

**why**: dpdm traces into node_modules and reports internal cycles within third-party packages. old versions of `helpful-errors` and `domain-objects` had internal circular imports. these came via `as-procedure@1.1.11` → `simple-log-methods@0.6.9` and `rhachet-artifact@1.0.3`.

**fix**: bumped upstream dependencies to versions with fixed cycles:
- `iso-time` 1.10.1 → 1.11.7 (has `helpful-errors@1.7.3`, `simple-log-methods@0.6.12`)
- `rhachet-artifact` 1.0.3 → 1.0.4 (has `domain-objects@0.31.13`, `helpful-errors@1.7.3`)
- `as-procedure` 1.1.11 → 1.1.13 (has `sdk-logs@0.9.2` instead of `simple-log-methods`, `helpful-errors@1.7.3`)

note: initially tried to exclude `node_modules` in `.dpdmrc.yaml` — this violates rule.forbid.dpdm-exclude-change. reverted and fixed properly via upstream bumps.
